# 針對這個差異化版本，畫出一個簡化架構圖（文字版），最好能和World Mini App 深度整合 + 多鏈部署，我之後可以拿去其他鏈使用。

先給你一個整體簡化版架構圖，接著分段說明每個元件和流程，整個設計都以「World Mini App 前端 + World ID 驗證 + CRE 私密結算 + 多鏈 payout」為核心。

***

## 高階元件列表

- World Mini App 前端（跑在 World App 裡的 web app，使用 MiniKit JS）[^1][^2]
- World ID 驗證層（IDKit + 你的 backend 驗 proof，產生匿名 user handle）[^3][^4]
- Prediction Market 控制合約（部署在 1 條「控制鏈」，建議 World Chain）
- 各目標鏈的「結算 / payout adapter 合約」（Base / Polygon / Arbitrum …，之後要新增鏈就多部署一個）
- Chainlink CRE Workflow（含 Confidential HTTP / Confidential Compute / multi-chain writes）[^5][^6][^7]
- 私密下注資料庫（只存在 CRE 機密工作流裡，鏈上只看到 aggregate 資訊）

## Mini App \& World ID 互動層

### A. World Mini App 前端

- 使用官方 MiniKit JS / create-mini-app 模板，在 World App 中以 webview 形式運行，並取得原生功能：sign tx、sendTransaction、verify 等。[^8][^2][^1]
- 前端主要職責：
    - 顯示市場列表 / 場景（由 backend 提供，僅是「問題、選項、賠率估計」等不敏感資訊）。
    - 呼叫 World ID 驗證流程（MiniKit + IDKit / World ID 4.0 路線）。[^4][^3]
    - 將「經過驗證的 session / user handle + 下注資料」發送到你的 backend API。


### B. World ID 驗證 Backend

- 借用 IDKit / World ID 雲端驗證流程：
    - Mini App 觸發 verify → 使用者在 World App 中 headless 完成 proof。
    - 前端將 proof 丟給你的 backend `/api/worldid/verify`。
    - backend 呼叫 World 的驗證端點（Server side），確認 proof 有效後：
        - 產生一個匿名 `userId`（例如對 nullifierHash 再做 hash），只在系統內使用，不回推任何 PII。
        - 把這個匿名 `userId` 存在 session / JWT，讓後續下注請求可以「綁定人」但不洩漏實際識別。

## 控制鏈合約層（World Chain / 任一主控鏈）

### C. Prediction Market 控制合約（部署一條主控鏈）

可以在 World Chain 或你比較熟的一條鏈上部署，建議：

- `MarketRegistry`
    - 建立/關閉市場（問題描述、選項、截止時間、目標結算鏈列表等）。
    - 儲存市場 metadata，但**不記每個人下注細節**。
- `BetIngress`
    - 使用者每次「下注」時，其實是：
        - 前端 → backend：送出下注請求 (marketId, option, amount, chainPreference, currency…)
        - backend 幫使用者用 MiniKit 發一筆 tx 到 `BetIngress`：
            - 鏈上只記：`event BetPlaced(marketId, hashedUserId, amount, optionId, nonce)`
            - `hashedUserId` 是你 backend 生成的匿名 id（再 hash 一次），無法直接反推真實 World ID。

`BetIngress` 合約的重點：

- 給 CRE workflow 一個「可監聽、可驗證的觸發點」。
- 不儲存完整下注明細（實際分配在哪個鏈、具體倉位等，留給機密層）。

***

## CRE 機密工作流層（私密下注 \& 多鏈結算）

### D. CRE Workflow（核心邏輯）

在 Chainlink CRE 上實作一條 workflow，職責分成兩大塊：[^9][^6][^7][^5]

1. **下注處理（Placement Path）**
    - Trigger：監聽控制鏈 `BetIngress` 的 `BetPlaced` event。
    - Step1：透過 Confidential HTTP 去 call 你的 backend API（或直接從 event 取得基礎資料），把下注請求補齊（幣種、使用者偏好鏈、預估賠率等）。
    - Step2：在 Confidential Compute 內部更新「該市場的私密 order book / position state」，例如：
        - `positions[marketId][hashedUserId] += amount (optionId)`
        - 這些結構只存在於 TEE 內記憶體/密文儲存，鏈上看不到。
    - Step3：如有需要，CRE 可以馬上在對應「目標鏈」上做 hedge / AMM 操作（例如幫整個市場在 Base 上的對應合約做流動性再平衡），但外界看不到是哪個人下的單，只看到 aggregate 流量。
2. **市場結算（Resolution Path）**
    - Trigger：
        - 主辦方 / oracle 提交「事件結果」到控制鏈 `MarketRegistry`，或
        - CRE 自己定期 poll 官方 API (Confidential HTTP) 得到結果。
    - Step1：結果進入 Confidential Compute TEE，和內部私密 `positions` 匹配，計算每個 `hashedUserId` 的 payout。
    - Step2：按使用者選的 payout 鏈，將 aggregate 結算金額寫入各鏈的 `PayoutAdapter` 合約：
        - 例如在 Base 上 call `PayoutAdapter.recordPayout(hashedUserId, marketId, amount)`。
    - Step3：TEE 內部保留 minimal audit log（也可產生 attestation 提供給審計者），然後清理掉已結算市場的明細。

## 多鏈 payout adapter 合約層

### E. 各鏈的 PayoutAdapter

在你想支援的每條鏈（Base / Polygon / Arbitrum / Scroll …）部署一個超輕量合約：

- `PayoutAdapter` 功能：
    - 接收來自 CRE workflow 的「最終 payout 指令」。
    - 記錄：`mapping(hashedUserId => mapping(marketId => amount))`。
    - 使用者透過 Mini App 選擇領錢的鏈，前端去對該鏈的 `PayoutAdapter.claim(marketId)`：
        - 合約根據 `hashedUserId`（透過 backend 簽名的 message 帶進來）驗證後，發送代幣。
- 由於 payout 指令是 CRE 發的，你可以：
    - 用 CCIP/CRE 的 multi-chain write 能力，確保是由可信的 Workflow DON 發出。[^6][^5]
    - 或在合約內驗證一組專用 signer（只給 CRE workflow 使用）。

## 交易流程分解（使用者角度）

### 1. 下單流程

1. 使用者在 World App 打開你的 Mini App（透過 MiniKit + 官方模板）。[^2][^1]
2. 若尚未驗證 World ID，前端觸發 verify：
    - World App 內 headless 完成 proof，回傳給前端，再送給 backend 驗證。[^3][^4]
3. backend 驗證成功 → 建立 session，內含匿名 `userId`。
4. 使用者選擇某個市場 \& 選項 \& 金額 \& payout 鏈偏好 → 前端呼叫 backend：`/api/bet`。
5. backend 透過 MiniKit 的 `sendTransaction` 幫他在控制鏈 `BetIngress` 上送出 tx：只包含 hashedUserId / marketId / amount / optionId。
6. `BetPlaced` event 觸發 CRE workflow → 在 TEE 裡更新私密 order book / positions。

### 2. 結算流程

1. 市場結束後，由你或某外部 oracle 在控制鏈上提交結果。
2. CRE workflow 監聽到結果，進入 Resolution Path：
    - 在 TEE 裡以結果 + 私密 positions 計算每位 `hashedUserId` 應得的 payout。
3. CRE 根據每人選的 payout 鏈，對應鏈的 `PayoutAdapter` 寫入 payout 記錄。
4. 使用者在 Mini App 的「我的倉位」頁面選擇領錢：
    - 前端向 backend 要一個「claim 簽名」，帶著 hashedUserId / marketId。
    - 前端透過 MiniKit 在對應鏈發一筆 tx 到 `PayoutAdapter.claim` 領回資金。

## 多鏈擴展與「帶去其他鏈」的方式

這個設計天然就是「控制層 + 多 payout 鏈」架構，之後要拿去別的鏈時，只要：

- 保持核心邏輯在 CRE workflow 和 TEE 內（鏈無關）。
- 對每條新鏈：
    - 部署一個新的 `PayoutAdapter`（邏輯完全一樣，只是鏈不同）。
    - 在 CRE workflow 配置多一個目標鏈的寫入 Capability。
- 若你未來想把控制層從 World Chain 搬到其他 L2，只要：
    - 重新部署 `MarketRegistry` + `BetIngress`，
    - 更新 CRE workflow 的 event source，
    - Mini App 端更新 RPC / chainId 就好，前端結構基本不變。[^10][^6]
