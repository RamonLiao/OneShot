# 好

那我直接幫你拆到可以開工的程度：合約要長怎樣、CRE workflow 步驟、以及 backend / Mini App 的 API 介面。你可以照這個切工分給前端 / 合約 / workflow。

***

## 一、合約設計（World Chain + 多鏈 Payout）

### 1. 控制鏈（建議 World Chain）合約

**(1) MarketRegistry.sol**

功能：管理市場生命週期，只存 metadata，不存個人下注。

關鍵資料結構：

```solidity
struct Market {
    uint256 id;
    string question;
    string[] options;
    uint256 closeTime;
    uint256 settleTime;
    bool settled;
    uint8 winningOption; // 0 = 未結算
    address creator;
}
```

主要函式：

- `createMarket(string question, string[] options, uint256 closeTime, uint256 settleTime, address[] payoutChains)`
- `closeMarket(uint256 marketId)`（可選，看你要不要硬關）
- `setResult(uint256 marketId, uint8 winningOption)` ← 由你／oracle／CRE 設定結果
- `getMarket(uint256 marketId)` view

事件：

```solidity
event MarketCreated(uint256 indexed marketId, string question);
event MarketClosed(uint256 indexed marketId);
event MarketSettled(uint256 indexed marketId, uint8 winningOption);
```


***

**(2) BetIngress.sol**

功能：作為「所有下注的 on-chain 入口」，只發事件，不存完整明細。

資料結構（如果要基本防 double bet，可以加個 mapping）：

```solidity
mapping(bytes32 => mapping(uint256 => bool)) public hasBet; 
// hashedUserId => marketId => bool
```

主要函式：

```solidity
function placeBet(
    uint256 marketId,
    bytes32 hashedUserId,
    uint8 optionId,
    uint256 amount,
    uint8 payoutChainId // 你自己定義 internal code, ex: 1=Base,2=Polygon
) external;
```

事件（CRE 的 trigger 來源）：

```solidity
event BetPlaced(
    uint256 indexed marketId,
    bytes32 indexed hashedUserId,
    uint8 optionId,
    uint256 amount,
    uint8 payoutChainId
);
```

邏輯重點：

- `placeBet` 只做基礎檢查（市場是否未關閉、金額>0 等），不持有或分配資金（真實資金流可交給 TEE / adapter / external escrow）。
- 事件 `BetPlaced` 是 CRE workflow 在控制鏈上的主要監聽來源。[^1]

***

### 2. 各目標鏈 PayoutAdapter 合約

在每條目標鏈部署一份 `PayoutAdapter.sol`，邏輯一樣，參數不同。

資料結構：

```solidity
mapping(bytes32 => mapping(uint256 => uint256)) public claimable; 
// hashedUserId => marketId => amount

mapping(bytes32 => mapping(uint256 => bool)) public claimed;
address public trustedPayoutSigner; // 或改成驗 CRE DON 的簽章
```

主要函式：

- `recordPayout(bytes32 hashedUserId, uint256 marketId, uint256 amount)`
    - `onlySigner` 修飾器，限制只能 CRE workflow / CCIP router 來叫。
- `claim(uint256 marketId, bytes calldata backendSig)`
    - 前端透過 backend 幫忙產一個 EIP‑712 簽名，內含 `hashedUserId`, `marketId`, `chainId`；
    - 合約 verify 這個簽名來確認呼叫者對應的 `hashedUserId`，防止別人盜領。

事件：

```solidity
event PayoutRecorded(bytes32 indexed hashedUserId, uint256 indexed marketId, uint256 amount);
event Claimed(bytes32 indexed hashedUserId, uint256 indexed marketId, address to, uint256 amount);
```

## 二、CRE Workflow 設計（Pseudo-code）

以 TypeScript workflow 為例，結構參考 CRE 官方範例：event trigger → 處理 → evm.write。[^2][^3][^1]

### 1. 下注路徑（Bet Path）

Trigger：監聽控制鏈 `BetIngress.BetPlaced` event。

Pseudo-code：

```ts
// Trigger handler
const onBetPlaced = async (runtime: Runtime<Config>, payload: EvmEventPayload) => {
  const event = parseBetPlacedEvent(payload); // marketId, hashedUserId, optionId, amount, payoutChainId

  // Step 1: 更新 TEE 內部狀態
  await positionsStore.update((state) => {
    state.addPosition({
      marketId: event.marketId,
      user: event.hashedUserId,
      optionId: event.optionId,
      amount: event.amount,
      payoutChainId: event.payoutChainId,
    })
  })

  // Step 2: Optional – 依 aggregate 狀態去外部預測市場 / AMM 對沖
  // const price = await httpClient.get("some-odds-api")
  // await evmClient.write({ chainSelector: BASE_CHAIN, ... })

  runtime.log(`Bet registered privately for user ${event.hashedUserId} on market ${event.marketId}`)
}
```

關鍵點：

- `positionsStore` 是你在 CRE 機密環境裡維護的 state，外部看不到。
- 若要控制 gas 和複雜度，初版可以不做對沖，只做「私密紀錄 + 最終結算」。

***

### 2. 結算路徑（Resolution Path）

Trigger：

- 監聽 `MarketRegistry.MarketSettled` event，或
- 定期 cron job 讀取某個結果 API 判斷市場該結算。[^1]

Pseudo-code：

```ts
const onMarketSettled = async (runtime: Runtime<Config>, payload: EvmEventPayload) => {
  const { marketId, winningOption } = parseMarketSettledEvent(payload)

  // Step 1: 從 TEE 內部 positionsStore 取出該市場所有頭寸
  const positions = await positionsStore.getByMarket(marketId)

  // Step 2: 計算每個 user 的 payout
  const payoutByUserAndChain = aggregatePayouts(positions, winningOption)

  // Step 3: 依不同 payoutChainId，分別對各鏈的 PayoutAdapter 寫入
  for (const [chainId, entries] of Object.entries(payoutByUserAndChain)) {
    const chainSelector = CHAIN_SELECTOR_MAP[chainId] // for CRE/CCIP[web:48][web:49]

    for (const entry of entries) {
      await evmClient.write({
        chainSelector,
        contractAddress: PAYOUT_ADAPTER_ADDRESS[chainId],
        abi: PayoutAdapterAbi,
        functionName: "recordPayout",
        args: [entry.hashedUserId, marketId, entry.amount],
      })
    }
  }

  // Step 4: 清掉已結算市場在 TEE 內的細節資料
  await positionsStore.clearMarket(marketId)

  runtime.log(`Market ${marketId} settled privately, payouts recorded on destination chains`)
}
```

這段用到 CRE 的典型 pattern：**event trigger → 讀取私密 state → 多鏈 write**，跟官方 tokenization / PoR demo 類似，只是你換成 prediction state。[^4][^2][^1]

## 三、Backend API 規格

Backend 主要三塊：World ID 驗證、下注 API、領錢 claim 輔助。

### 1. World ID 驗證

`POST /api/worldid/verify`

Request：

```json
{
  "proof": { "merkle_root": "...", "nullifier_hash": "...", "proof": "...", "signal": "..." }
}
```

步驟：

1. Backend 將 proof 丟到 World 的 verify 端點（或用官方 SDK）。[^5]
2. 驗證成功後：
    - 從 `nullifier_hash` 衍生出 `userId`：`userId = keccak256(nullifier_hash || appId)`。
    - 產生 session / JWT：`{ userId, hashedUserId = keccak256(userId || salt) }`。
3. 回傳：
```json
{ "status": "ok", "hashedUserId": "0x..." }
```


***

### 2. 下單 API

`POST /api/bet`

Headers：帶上 session / JWT（內含 hashedUserId）。

Request：

```json
{
  "marketId": 1,
  "optionId": 0,
  "amount": "1000000",
  "payoutChainId": 1
}
```

步驟：

1. Backend 驗 session，取得 `hashedUserId`。
2. 準備呼叫 `BetIngress.placeBet()` 的交易資料：
    - address = `BetIngress` on World Chain
    - function = `"placeBet"`
    - args = `[marketId, hashedUserId, optionId, amount, payoutChainId]`
3. 用 MiniKit 的 `sendTransaction` 叫使用者簽送這筆 tx（前端會用 JS 啟動）。[^6][^7]
4. 回傳 transaction id 給前端，用於追蹤狀態。

Response：

```json
{
  "status": "sent",
  "transactionId": "txid-from-minikit"
}
```


***

### 3. 查詢市場 / 我的倉位

- `GET /api/markets` → 回傳公開的 markets + 大致盤口資訊（從控制鏈 + TEE aggregate 資料）。
- `GET /api/positions` → 用 session 中的 `hashedUserId` 去 TEE / DB 問「這個 user 在每個市場下注了多少」，只回自己看得到的聚合資訊。

***

### 4. 領錢 claim API 輔助

因為 `hashedUserId` 只在 backend 知道 user 對應誰，所以 claim 時要 backend 幫忙簽一個 message。

`POST /api/payout/prepare-claim`

Request：

```json
{
  "marketId": 1,
  "payoutChainId": 1
}
```

步驟：

1. Backend 驗 session，拿到 `hashedUserId`。
2. 呼叫對應鏈的 RPC（或直接讓使用者自己查 UI，但簡化先假設 backend 查）：
    - 讀 `PayoutAdapter.claimable(hashedUserId, marketId)`。
3. 若 `amount > 0` 且未 claimed：
    - 對以下資料做 EIP‑712 簽名：
        - `domain: { name: "PayoutAdapter", chainId, verifyingContract }`
        - `types: Claim { hashedUserId, marketId, amount, deadline }`
4. 回傳：
```json
{
  "status": "ok",
  "amount": "1000000",
  "deadline": 1710000000,
  "signature": "0x..."
}
```

前端再用 MiniKit `sendTransaction` 對目標鏈的 `PayoutAdapter.claim(marketId, signature)` 發 tx。

## 四、World Mini App 前端關鍵點（超簡版 flow）

根據 MiniKit sendTransaction 文件，一般 pattern 是：組好合約呼叫 → 交給 MiniKit.[^8][^6]

Pseudo-code（React）：

```ts
import { MiniKit } from '@worldcoin/minikit-js'
import BetIngressAbi from './abi/BetIngress.json'

const placeBet = async (marketId, optionId, amount, payoutChainId) => {
  // 1. 叫 backend 確認 session/hashedUserId 並拿到 tx 參數(可選)
  const res = await fetch('/api/bet-intent', { ... })

  // 2. 透過 MiniKit 發送交易到 World Chain 的 BetIngress
  const { payload } = await MiniKit.commandsAsync.sendTransaction({
    transaction: [
      {
        address: BET_INGRESS_ADDRESS,
        abi: BetIngressAbi,
        functionName: 'placeBet',
        args: [res.marketId, res.hashedUserId, optionId, res.amount, payoutChainId],
      },
    ],
  })

  // 3. 記錄 transactionId 用 useWaitForTransactionReceipt 追蹤
}
```
